- 생성 파일 경로:

```text

```

- 다만 Android 설정은 현재 정식 배포용 서명이 아니라 debug signing 설정을 사용 중이다.

## 2. Dead Reckoning 코드 존재 여부 확인

- 프로젝트 검색 결과 `dead reckoning` 계산 로직은 이미 존재했다.
- 관련 파일:
  - `lib/services/dead_reckoning_calculator.dart`
  - `lib/pages/indoor_navigation_page.dart`
- 당시 상태는 계산기 클래스는 있었지만, 실제 화면에서 계산기를 호출하지 않고 센서 raw 값만 표시하고 있었다.

## 3. 시작 위치 `(0, 0)` 기준 위치/방향 표시 연결

- `IndoorNavigationPage`에서 센서 스트림을 받아 `DeadReckoningCalculator`에 전달하도록 연결했다.
- 시작 위치를 `(0, 0)`으로 두고 아래 정보를 화면에 표시하도록 변경했다.
  - 현재 좌표 `(x, y)`
  - 방향 라벨 `N/NE/E/...`
  - 방향 각도
  - 걸음 수
- 이 단계에서는 기본적인 시각화 연결이 목적이었다.

## 4. 초기 step detection 문제 수정

- 사용 중 문제:
  - 걸음 수가 거의 늘지 않음
  - 좌표가 잠깐만 바뀌고 멈춤
- 원인:
  - `accelerometer` 처리 방식과 threshold가 실제 모바일 센서값에 잘 맞지 않았다.
- 수정:
  - 한때 `userAccelerometer` 기반으로 바꾸고 threshold를 낮춰 반응성을 높였다.
  - 총 이동 거리 표시를 추가했다.
- 이후 GitHub 구현 비교 과정에서 입력 정의를 다시 raw accelerometer 기준으로 조정했다.

## 5. `Aymdi/DeadReckoning` 구현과 비교

- 사용자가 제시한 저장소:
  - `https://github.com/Aymdi/DeadReckoning`
- 비교 결과:

- 차이점으로 확인된 항목:
  - step 계산 방식
  - heading 계산 방식
  - 센서 입력 처리 방식

## 6. `Aymdi/DeadReckoning` 스타일 1차 반영

- 원본 흐름에 맞추기 위해 계산기를 재구성했다.
- 반영 내용:
  - `threshold crossing` 기반 step 계산 구조 도입
  - crossing 2회를 1걸음으로 환산하는 상태값 추가
  - raw accelerometer norm 기반 처리
  - magnetometer/gyroscope를 이용한 heading 보정 경로 추가
- 이 단계에서 원본 저장소 구조에는 더 가까워졌지만, 실시간 환경에 맞추기 위해 일부는 러닝 평균/실시간 필터 방식으로 바뀌었다.

## 7. Step Detection 2차 개선

- 사용자가 "2번도 적용" 요청 후 step detection을 더 정교화했다.
- 반영 내용:
  - 동적 threshold 계산 추가
  - hysteresis 적용
  - `상승 crossing + 하강 crossing` 한 주기를 걸음으로 인식
  - 디버그 확인용 값 추가
    - `Crossings`
    - `Filtered`
    - `Threshold`
- 이 단계에서 걸음 검출 로직은 이전보다 덜 흔들리고, 현장에서 threshold 반응을 눈으로 볼 수 있게 되었다.

## 8. Magnetometer/Heading drift 문제 확인

- 사용자가 "가만히 있어도 Magnetometer가 계속 움직인다"는 문제를 제기했다.
- 확인 결과 최근 변경 중 아래 부분이 영향을 줄 가능성이 컸다.
  - raw gyroscope 적분 누적
  - magnetometer 평균 제거
  - 직접 회전 보정
- 이 부분은 정지 상태에서도 drift를 만들 수 있어 현재 증상과 관련성이 높다고 판단했다.

## 9. `WiseLabCMU/IMUPlot` 구현과 비교

- 사용자가 제시한 저장소:
  - `https://github.com/WiseLabCMU/IMUPlot`
- 비교 결과:
  - 현재 구현과 상당히 달랐다.
  - `IMUPlot`은 raw gyro 적분 대신 Android의 다음 센서를 사용한다.
    - `TYPE_GEOMAGNETIC_ROTATION_VECTOR`
    - `TYPE_GAME_ROTATION_VECTOR`
  - 초기 몇 걸음 동안 heading offset을 보정하는 구조를 사용한다.
  - step detection은 moving average + fixed threshold 방식이다.

## 10. `IMUPlot` 스타일 heading 적용

- 사용자가 "1번부터 하나씩" 요청하여 heading 계산부터 `IMUPlot` 방식에 가깝게 교체했다.
- 반영 내용:
  - Android 네이티브 `MainActivity.kt`에 EventChannel 추가
  - 아래 rotation vector 센서 azimuth를 Flutter로 전달
    - geomagnetic rotation vector
    - game rotation vector
  - Flutter에서 해당 값을 받아 계산기에 전달
  - 계산기에서 `geomagnetic azimuth - game azimuth` 기반 offset을 누적해 초기 방향을 보정
  - rotation-vector 값이 있으면 이 heading 경로를 우선 사용하고, 없을 때만 기존 fallback을 사용
- 기대 효과:
  - raw gyro 적분 기반 drift 감소
  - 정지 상태 heading 안정성 개선

## 14. 현재 남아 있는 참고 사항

- APK는 정상 빌드되지만 release signing은 아직 debug signing 설정이다.
- step detection은 여러 번 개선했지만 실제 기기/휴대 방식에 따라 threshold 튜닝이 더 필요할 수 있다.
- heading은 현재 `IMUPlot` 스타일의 rotation-vector 경로를 우선 적용한 상태이며, 실제 기기에서 drift가 충분히 줄었는지 추가 확인이 필요하다.
